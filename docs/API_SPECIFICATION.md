# Aetheris API Specification

## 1. Purpose

This document defines the public API contracts, endpoint conventions, authentication rules, error standards, pagination behavior, and request/response semantics for Aetheris API.

The API is designed to be:

- REST-oriented
- versioned
- predictable
- tenant-aware
- machine-consumable
- externally integratable
- stable over time

The OpenAPI schema generated by FastAPI is the machine-readable source of truth, while this document defines architectural standards and behavioral expectations.

---

# 2. API Design Principles

## 2.1 Versioning

All endpoints must be versioned.

Base prefix:

    /api/v1

Breaking changes require a new version namespace.

---

## 2.2 Content Type

Default request and response format:

    application/json

File uploads use:

    multipart/form-data

---

## 2.3 Authentication Methods

Supported authentication methods:

- JWT Bearer tokens
- Organization-scoped API keys

---

## 2.4 Multi-Tenant Enforcement

Every request must execute within a tenant context.

Tenant context may derive from:

- authenticated user organization membership
- API key ownership

Cross-tenant access must always return authorization failures.

---

## 2.5 Time Format

All timestamps use:

- UTC
- ISO 8601 format

Example:

    2026-05-26T14:20:00Z

---

## 2.6 ID Strategy

Primary resource identifiers use UUIDs.

Examples:

- user_id
- organization_id
- document_id
- job_id
- chunk_id

---

# 3. Authentication & Authorization

## 3.1 JWT Authentication

JWT access tokens authenticate users.

Access tokens should remain short-lived.

Recommended duration:

- 15–30 minutes

Refresh tokens are used for session continuation.

---

## 3.2 Refresh Tokens

Refresh tokens:

- rotate on use
- can be revoked
- are tenant-aware
- must be stored securely

---

## 3.3 API Keys

API keys are organization-scoped.

Characteristics:

- visible only once at creation
- stored hashed
- revocable
- rate-limited
- auditable

Recommended format:

    athr_live_xxxxxxxxx

---

## 3.4 Authorization Model

Authorization is role-based.

Initial roles:

- owner
- admin
- member

---

# 4. Standard Response Structure

## 4.1 Success Response Pattern

Example structure:

    {
      "success": true,
      "data": {},
      "meta": {}
    }

---

## 4.2 Error Response Pattern

All API errors must follow a consistent structure.

Example:

    {
      "success": false,
      "error": {
        "code": "DOCUMENT_NOT_FOUND",
        "message": "Document does not exist.",
        "request_id": "uuid"
      }
    }

---

## 4.3 Request Correlation

Every request must include:

- request_id
- trace correlation metadata internally

Request IDs should appear:

- in logs
- in error responses
- in traces

---

# 5. Pagination Standards

## 5.1 Pagination Model

Use cursor-based or limit/offset pagination consistently.

MVP recommendation:

- limit
- offset

---

## 5.2 Pagination Parameters

Supported parameters:

- limit
- offset

Defaults:

- limit: 20
- maximum limit: 100

---

## 5.3 Pagination Response Metadata

Example:

    {
      "success": true,
      "data": [],
      "meta": {
        "limit": 20,
        "offset": 0,
        "total": 154
      }
    }

---

# 6. Rate Limiting

## 6.1 General Policy

All routes must enforce rate limiting.

Limits may vary by:

- user tier
- API key
- IP address

---

## 6.2 Default MVP Limit

Recommended MVP baseline:

- 100 requests/minute per API key

---

## 6.3 Rate Limit Headers

Recommended headers:

- X-RateLimit-Limit
- X-RateLimit-Remaining
- X-RateLimit-Reset

---

# 8. Authentication Endpoints

Endpoint:

    POST /api/v1/auth/login

- authenticate user
- issue tokens

      "success": true,
      "data": {
        "access_token": "jwt",
        "refresh_token": "jwt",
        "expires_in": 1800
      }
    }

Endpoint:

    POST /api/v1/auth/refresh

Purpose:

- rotate access tokens

## 8.3 Logout

Endpoint:

# 11. Document Endpoints

## 11.1 Upload Document

Endpoint:

    POST /api/v1/documents

Content-Type:

    multipart/form-data

Supported formats:

- PDF
- TXT
- MD

Behavior:

- validates upload
- stores file
- creates processing job
- returns async job metadata

Response example:

    {
      "success": true,
      "data": {
        "document_id": "uuid",
        "job_id": "uuid",
        "status": "QUEUED"
      }
    }

---

## 11.2 List Documents

Endpoint:

    GET /api/v1/documents

Purpose:

- list tenant documents

Supported filters:

- status
- created_at
- filename

---

## 11.3 Retrieve Document

Endpoint:

    GET /api/v1/documents/{document_id}

Purpose:

- retrieve document metadata

---

## 11.4 Delete Document

Endpoint:

    DELETE /api/v1/documents/{document_id}

Purpose:

- soft-delete document
- trigger vector cleanup

---

Endpoint:

    POST /api/v1/search

- semantic retrieval without generation

      "query": "What is the refund policy?",
      "top_k": 5
    }

    {

      "data": {
        "results": [

            "chunk_id": "uuid",
            "document_id": "uuid",
            "score": 0.91,
            "text": "..."
          }
        ]
      }
    }

---

## 13.2 Search Filters

Supported filters may include:

- document_id
- tags
- created_at
- metadata filters

---

# 14. RAG Generation Endpoints

## 14.1 Ask Endpoint

Endpoint:

    POST /api/v1/ask

Purpose:

- retrieval-augmented answer generation

Request example:

    {
      "question": "What are the onboarding requirements?",
      "top_k": 5
    }

Response example:

    {
      "success": true,
      "data": {
        "answer": "Generated grounded answer.",
        "citations": [],
        "retrieved_chunks": []
      }
    }

---

## 14.2 RAG Principles

The generation system must:

- use retrieved context only
- include citations
- avoid unsupported claims
- prefer uncertainty over hallucination

# 16. Error Standards

## 16.1 Common Error Codes

Examples:

- INVALID_CREDENTIALS
- ACCESS_DENIED
- DOCUMENT_NOT_FOUND
- RATE_LIMIT_EXCEEDED
- INVALID_UPLOAD
- JOB_FAILED
- INTERNAL_ERROR

---

## 16.2 Validation Errors

Validation failures should return:

- explicit field-level detail
- stable structure
- machine-readable semantics

---

## 16.3 Security Errors

Authentication failures must avoid leaking:

- existence of accounts
- internal implementation details
- tenant-sensitive information

---

- adding fields

- version increment

# 20. API Design Invariants

The following are mandatory invariants:

- every request is authenticated unless explicitly public
- every tenant boundary is enforced
- every async workflow is observable
- every error response is structured
- every public endpoint is versioned
- every critical operation is auditable
