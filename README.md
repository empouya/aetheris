# Aetheris API

Aetheris API is a headless, enterprise-oriented backend for document ingestion, semantic retrieval, and retrieval-augmented generation (RAG). It is designed as a B2B platform that ingests unstructured documents, converts them into searchable semantic representations, and exposes developer-first API endpoints for search and grounded answer generation.

## What Aetheris is for

Aetheris solves the “dark data” problem in enterprises: large volumes of PDFs, text files, and markdown documents that are difficult to search with traditional keyword systems. The platform processes documents asynchronously, extracts text, chunks content, generates embeddings, stores vectors, and serves search and answer endpoints with citations.

## High-level architecture

Aetheris uses a modular monolith architecture with asynchronous workers.

The application is split into domain-oriented modules such as:

- auth
- organizations
- api_keys
- documents
- ingestion
- processing
- embeddings
- retrieval
- generation
- billing
- observability

The public API layer is intentionally thin. Business logic lives in services. Persistence access is abstracted through repositories. Background work is executed through Celery workers.

## How the system works

### Upload flow

1. A client authenticates with a JWT or API key.
2. The client uploads a file through the API.
3. The API validates the request, stores metadata, uploads the file to object storage, and creates a processing job.
4. A worker processes the job asynchronously.
5. The document is extracted, cleaned, chunked, embedded, and indexed in Qdrant.
6. The job transitions to a terminal state when complete or failed.

### Search flow

1. The client submits a natural-language query.
2. The query is embedded using the configured embedding provider.
3. Qdrant returns the most relevant chunks, filtered by tenant and optional metadata.
4. The API returns matching chunks and citations.

### Ask flow

1. The client submits a natural-language question.
2. A retrieval pass identifies the best supporting chunks.
3. The system assembles a bounded context window.
4. The configured LLM provider generates a grounded answer.
5. The API returns the answer, citations, and retrieved chunks.

## Multi-tenancy model

Aetheris is multi-tenant from day one.

The tenancy model is based on:

- organizations as the top-level boundary
- users belonging to organizations
- API keys scoped to organizations
- documents, jobs, chunks, queries, and usage records scoped to organizations

All access checks must enforce tenant boundaries explicitly.

## Data storage

Aetheris uses:

- PostgreSQL for users, organizations, documents, jobs, query logs, usage, and audit records
- Qdrant for semantic vector storage
- object storage for original files and derived extraction artifacts

The system stores both raw documents and derived artifacts so that documents can be reprocessed, audited, and debugged later.

## Observability and reliability

Aetheris is designed with structured JSON logging and OpenTelemetry-compatible trace propagation. The platform tracks:

- request identifiers
- tenant identifiers
- job identifiers
- trace identifiers

Operational reliability is based on:

- asynchronous processing
- retryable jobs
- idempotent worker tasks
- explicit job failure states
- health and readiness checks
- bounded resource usage
- graceful degradation on provider failure

## Security principles

Aetheris is designed around:

- HTTPS-only production traffic
- JWT access tokens with rotating refresh tokens
- hashed API keys
- strong tenant isolation
- rate limiting
- input validation
- secure logging
- secrets never committed to the repository

## Local development

The local environment is Docker Compose centered. A typical development stack includes:

- API service
- worker service
- PostgreSQL
- Redis
- Qdrant
- MinIO
- Nginx

This keeps local development close to production architecture while remaining easy to run and debug.

## Development philosophy

Aetheris is built using a modular monolith and vertical-slice delivery.

The intended development style is:

- build foundations first
- implement one complete workflow at a time
- keep pull requests small
- write tests with each meaningful change
- keep architecture documentation aligned with code
- avoid premature microservices, Kubernetes, and speculative optimization
